# 📚 Documentation

Okuri 프로젝트의 모든 문서를 한눈에 볼 수 있습니다.  
**문서를 읽는 순서**: VISION → SYSTEM → TASKS → DEVLOG

---

## 요약

```
docs/
├── VISION.md          # 프로젝트 비전, 핵심 기능 (Why)
├── SYSTEM.md          # 전체 아키텍처, 개요 스키마 (What)
├── TASKS.md           # 개발 타임라인, 체크리스트 (When)
├── DEVLOG.md          # 개발 일지 (기술적 의사결정, 트러블슈팅)
└── IDEAS.md           # 아이디어 메모장
└── feature/          # 기능별 미니 스펙 문서
```

## 📖 문서 목록

### 🎯 [VISION.md](./VISION.md)

> **제품의 방향성과 핵심 기능**

- 프로젝트 미션 및 목표
- MVP 핵심 기능 6가지 (가족 그룹, 피드, 블로그, 공지사항, 캘린더, 펑)
- Nice to Have (미래 기능)
- 디자인 원칙

**언제 보나요?**  
프로젝트의 본질과 "왜 이걸 만드는가?"를 알고 싶을 때

---

### 🏗 [SYSTEM.md](./SYSTEM.md)

> **기술 아키텍처와 구현 설계**

- 기술 스택
- 시스템 아키텍처 및 데이터 흐름
- 폴더 구조 및 파일 명명 규칙
- 데이터베이스 스키마 (예상)
- 핵심 기술 챌린지 (펑 콘텐츠 수명 관리, 캐시 최적화 등)
- UI 네비게이션 구조

**언제 보나요?**  
코드 작성 전 구조를 이해하고 싶을 때, 새로운 기능을 설계할 때

**⚠️ 주의**: 코드가 변경될 때마다 이 문서도 함께 업데이트해야 합니다.

---

### ✅ [TASKS.md](./TASKS.md)

> **프로젝트 계획 및 진행 상황**

- 개발 타임라인 (Phase별)
- Milestone별 작업 목록 및 완료 여부
- 커밋 컨벤션 및 운영 원칙
- Backlog (추후 작업)

**언제 보나요?**  
지금 무엇을 해야 하는지, 다음에 뭘 할지 확인할 때

**매일 업데이트**: 작업 완료 시 체크박스 ✅ 표시

## 🗂 문서 작성 원칙

### 1. 각 문서는 명확한 목적을 가진다

- **VISION**: 무엇을, 왜 만드는가?
- **SYSTEM**: 어떻게 구현하는가?
- **TASKS**: 언제, 무엇을 하는가?
- **DEVLOG**: 무슨 일이 있었는가?
- **IDEAS**: 앞으로 뭘 할까?
- **NOTES**: 잊지 말아야 할 것은?

### 2. 살아있는 문서로 유지

- 코드 변경 시 문서도 업데이트
- 작업 완료 시 TASKS 체크
- 문제 해결 시 DEVLOG 기록

### 3. 나중에 README.md로 통합

- 프로젝트 완성 후 핵심 내용만 추출
- 루트의 README.md에 프로젝트 소개 작성
